# Personal web prototype — v0.15

Open `index.html` from a local server and enter the map through `map.html`.

Recommended:

```bash
cd personal_web_prototype
python -m http.server 8000
```

Then open:

```text
http://localhost:8000
```

## What changed in v0.15

This version adds the first serious data pipeline for the map:

- New script: `scripts/generate_solar_json.py`.
- New seed catalogues in `assets/data_sources/`:
  - `bright_stars_seed.csv`
  - `open_clusters_seed.csv`
  - `cluster_members_seed.csv`
- New Python requirements file: `requirements-data.txt`.
- The web map can now consume precomputed `X_pc, Y_pc, Z_pc` values when present in the JSON.
- Named cluster members generated by the pipeline get catalogue-based positions when SIMBAD resolves them.
- If a cluster member cannot be resolved, the script skips it by default so no named fake star is drawn. The demo-only `--allow-visual-fallback` option can still be used for visual tests.

The existing JSON files are still included so the map opens immediately. Running the generator replaces `assets/data/bright_stars.json`, `assets/data/systems.json` and the browser bundle `assets/js/solar-data.js` with fresh SIMBAD-derived data.

## Generate Gaia/SIMBAD-style JSON files

Install the data dependencies:

```bash
python -m pip install -r requirements-data.txt
```

Run the generator from the project root:

```bash
python scripts/generate_solar_json.py
```

Dry run without overwriting the map data:

```bash
python scripts/generate_solar_json.py --dry-run
```

The generator currently uses SIMBAD resolution for:

- bright field stars;
- open-cluster centres;
- named bright cluster members.

For named cluster members, the script computes:

```text
member offset_pc = member Galactic Cartesian XYZ - cluster Galactic Cartesian XYZ
```

These offsets are what the Three.js map uses to place the star inside its parent cluster.

## Important scientific caveat

This is now a proper catalogue-generation workflow, but it is not yet a rigorous Gaia cluster-membership selection.

The next data upgrade should be one of these:

1. feed the script with published Gaia membership tables for each open cluster; or
2. add a Gaia DR3 cone-query stage with proper astrometric cuts in parallax and proper motion.

Only after that should the cluster morphologies be treated as scientifically meaningful rather than atlas-style visual reconstructions.

## v0.13 data note

The Pleiades model has been corrected so that the labelled bright members use curated J2000 sky coordinates projected at the cluster mean distance. This gives a realistic projected morphology for Alcyone, Atlas, Pleione, Electra, Maia, Merope, Taygeta, Celaeno and Asterope. The small unlabelled background stars inside each cluster remain visual/synthetic until a Gaia membership table is ingested.

The data generator now supports manual `ra_deg`, `dec_deg` and `distance_pc` columns in the seed CSV files. This means cluster members with curated coordinates can be converted to Galactic Cartesian `X_pc, Y_pc, Z_pc` even without live SIMBAD access. SIMBAD/Gaia ingestion remains the next step for fully catalogue-grade cluster populations.


## v0.14 data policy

This version stops pretending that unresolved cluster-member labels are real.

- Pleiades keeps its curated named members.
- Other clusters keep their real cluster centres and visual faint populations, but fake named bright-member markers have been removed until the SIMBAD/Gaia generation is run.
- `scripts/generate_solar_json.py` now resolves the named members listed in `assets/data_sources/cluster_members_seed.csv` through SIMBAD.
- By default, unresolved named members are skipped rather than drawn with fake offsets.

Run with internet access:

```bash
python -m pip install -r requirements-data.txt
python scripts/generate_solar_json.py
```

Demo-only fallback, not recommended for the scientific version:

```bash
python scripts/generate_solar_json.py --allow-visual-fallback
```

For the final scientific atlas, named bright members can come from SIMBAD, but the full internal cluster population should come from Gaia DR3 membership tables or a curated published membership catalogue.

## v0.15 cluster-member upgrade

This version prepares the “good” open-cluster workflow:

- `cluster_members_seed.csv` now contains named bright-star seeds for all current open clusters, not only the Pleiades.
- Pleiades keeps curated J2000 coordinates in the seed table so it works immediately offline.
- Hyades, Praesepe, α Per, IC 2391, IC 2602 and NGC 2451A now have SIMBAD query identifiers for their brightest named members.
- The generator records how many named seeds were resolved and how many are still pending, and the map shows that status in the left panel.
- Unresolved named stars are still not drawn as fake markers. This is deliberate. Once the generator is run with internet access and `astroquery`, those named members are resolved and inserted into the cluster JSON.
- Hot-star colours were shifted slightly bluer.

Run this from the project root to resolve the remaining clusters:

```bash
python -m pip install -r requirements-data.txt
python scripts/generate_solar_json.py
```

The result updates:

```text
assets/data/bright_stars.json
assets/data/systems.json
assets/js/solar-data.js
```

Important: SIMBAD is the right source for named bright members. It is not the final source for the full faint population of each cluster. That should later come from Gaia DR3 membership tables or a published membership catalogue.

## v0.17 — three brightest stars per constellation

This version adds a dedicated `constellationStars` map layer and a generator:

```bash
python scripts/generate_constellation_stars_from_hyg.py --download --radius-pc 300 --top-n 3
```

The generator uses the HYG catalogue fields `con`, `dist`, `mag`, `ra`, `dec`, `proper`, `bf`, `spect`, etc. It filters stars to `dist <= 300 pc`, groups by constellation, keeps the three lowest-V-magnitude stars per constellation, converts ICRS coordinates to Galactic Cartesian coordinates, and updates:

```text
assets/data/constellation_stars.json
assets/js/solar-data.js
```

You can also use a local HYG CSV file:

```bash
python scripts/generate_constellation_stars_from_hyg.py --hyg-csv path/to/hygdata_v41.csv
```

The layer is intentionally separate from the original iconic-bright-star layer, so we can switch it on/off without touching the curated cluster and field-star work.

## v1.0-pre — solar_map_400pc.py bridge

This version adds the first bridge from the thesis `solar_map_400pc.py` workflow into the interactive Three.js web atlas.

New files:

```text
scripts/solar_map_400pc.py
scripts/export_400pc_web_json.py
```

The exporter imports `solar_map_400pc.py`, uses its YAML/caches/resolvers, and writes the browser files:

```text
assets/data/bright_stars.json
assets/data/systems.json
assets/data/regions.json
assets/js/solar-data.js
```

Required next to `scripts/solar_map_400pc.py` before running:

```text
scripts/solar_map_400pc_data.yaml
```

Optional, if available in the project root, exactly as in the thesis script:

```text
cache/*.ecsv
handbook_distances.html
map3D_GAIAdr2_feb2019.h5
```

Run:

```bash
python -m pip install -r requirements-data.txt
python scripts/export_400pc_web_json.py --solar-map-py scripts/solar_map_400pc.py --radius-pc 400
```

By default, the exporter keeps only open clusters from the scientific 400 pc table. Associations can be added with:

```bash
python scripts/export_400pc_web_json.py --include-associations
```

The web renderer now also understands 3D dust/cloud blob groups exported from `_DUST_XY_BLOBS`: each 2D dust footprint is lifted into a translucent 3D ellipsoid-like volume while preserving the existing label/click style.

Important: the uploaded Python script loads its scientific dictionaries from `solar_map_400pc_data.yaml`, so the exporter cannot regenerate the final 400 pc dataset until that YAML file is placed beside the script.

## v1.0-dust-fix

The 400 pc bridge now preprocesses the scientific dust/cloud source files into browser JSON:

- `handbook_distances.html` is parsed by `scripts/export_400pc_web_json.py` and exported as 3D molecular-cloud bubbles in `assets/data/regions.json`.

## v1.0.3 — exporter fixes for stars and H5 dust

The 400 pc exporter now tries the cached `cache/bright_named_stars_400pc_v3.ecsv` first. If that cache exists from the original `solar_map_400pc.py`, it is used directly, avoiding repeated SIMBAD calls and giving the full curated bright-star sample. If the cache is missing, the exporter falls back to SIMBAD with more robust RA/Dec column handling.

H5 Lallement/STILISM dust is now exported as smooth connected components rather than merged into only a few very large clouds or plotted as individual voxel points. Useful controls:

```bash
python scripts/export_400pc_web_json.py --star-vmag-max 3.0
python scripts/export_400pc_web_json.py --h5-percentile 93.5 --max-dust-clouds 70
python scripts/export_400pc_web_json.py --no-resolve-simbad-stars
```

For the best bright-star layer, first generate/cache the bright-star ECSV with the original thesis script, then run the web exporter.

## v1.0.4 — SIMBAD bright-star export fix

The bright-star exporter now accepts both classic SIMBAD sexagesimal coordinates and newer decimal-degree RA/DEC table outputs. The previous parser silently discarded successful SIMBAD queries when RA/DEC were returned as decimal degrees, so the map could stay stuck at the original 29 web-curated stars even with `--star-vmag-max 3`.

The exporter now prints a small diagnostic such as:

```text
Bright-star SIMBAD: resolved N / M pending stars
```

so it is clear whether SIMBAD is resolving additional YAML stars.


## v1.0.5 — SIMBAD aliases for common bright-star names

The 400 pc exporter now includes explicit SIMBAD aliases for common names that may not resolve directly in SIMBAD: Cih, Girtab, Gienah Cyg and Nash. The exported labels remain the human-readable names, but the queries use robust Bayer/SIMBAD identifiers.


## v1.0.6 — all-sky bright-star query

The exporter no longer treats `BRIGHT_NAMED_STARS` in `solar_map_400pc_data.yaml`
as the full bright-star catalogue. That YAML list is only a curated label list
for the thesis figure.

By default, `scripts/export_400pc_web_json.py` now also performs a magnitude-limited
SIMBAD TAP query:

```bash
python scripts/export_400pc_web_json.py --star-vmag-max 3
```

This means: include all SIMBAD objects with V <= 3 and parallax distance <=
`--radius-pc` (400 pc by default), not only the named stars already present in
the YAML. Results are cached in `cache/simbad_allsky_bright_*.json`.

Useful options:

```bash
python scripts/export_400pc_web_json.py --star-vmag-max 3 --refresh-all-sky-star-cache
python scripts/export_400pc_web_json.py --star-vmag-max 4.2
python scripts/export_400pc_web_json.py --no-all-sky-simbad-stars
```


## v1.0.7 — SIMBAD TAP / SciPy warning fix

- Removed the top-level SciPy import from `scripts/export_400pc_web_json.py`; SciPy is now imported lazily and the known NumPy/SciPy compatibility warning is suppressed.
- Fixed the all-sky SIMBAD TAP ADQL query. The exporter now orders by the selected alias `vmag` rather than by `table.column`, which can fail in the SIMBAD TAP parser after joins.
- Added multiple SIMBAD schema variants for the V-band all-sky query (`allfluxes` and `mesFlux`) so the script is less dependent on the exact server/schema version.
- Use `--refresh-all-sky-star-cache` once after updating if an old empty cache exists.

## v1.0.9 — label tiers, star de-duplication, brighter clouds

- Curated/YAML/web bright stars now have priority over all-sky SIMBAD aliases. This prevents duplicates such as `Capella` / `* alf Aur`, `Sirius` / `* alf CMa`, `Alioth` / `* eps UMa`, etc.
- The all-sky SIMBAD layer skips objects already represented by the curated list, using canonical-name aliases and near-identical 3D positions.
- Field-star labels now depend on both distance and camera zoom:
  - full 400 pc view: labels mainly for 100--400 pc stars;
  - intermediate view: labels for 50--100 pc stars plus the brightest <50 pc stars;
  - close view: labels for <50 pc stars.
- Existing projected label behaviour while rotating the map is preserved.
- Dust/molecular/H5 clouds are rendered brighter and as larger soft volumetric clouds.
